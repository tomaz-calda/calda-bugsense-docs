## Overview

BugSense exposes a [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) server that lets ChatGPT query your projects, issues, and project health directly through natural language. Users authenticate with their BugSense credentials via an OAuth flow, and ChatGPT can then call BugSense tools on their behalf.

Access is scoped to the user's role — admins see all projects, clients only see their assigned project.

## How it works

<Steps>
  <Step title="Add BugSense to ChatGPT">
    In ChatGPT, add BugSense as a remote MCP server using the backend URL. ChatGPT discovers the OAuth and MCP endpoints automatically via `.well-known` metadata.
  </Step>
  <Step title="Authorize with your BugSense account">
    ChatGPT redirects to the BugSense OAuth authorize page. You sign in with your email and password, then grant ChatGPT read access to your projects and issues.
  </Step>
  <Step title="Ask questions">
    ChatGPT can now call BugSense tools to answer questions like "What are the high-priority bugs in my project?" or "Give me a health summary for this week."
  </Step>
</Steps>

## Available tools

ChatGPT can call these tools through the MCP protocol:

| Tool | Description |
|------|-------------|
| `list_projects` | List all BugSense projects with their name, team lead, and active issue count |
| `list_issues` | List issues for a project with filters (status, priority, platform, environment). Returns up to 50 issues |
| `get_issue` | Get full details of a specific issue including its entire comment thread. Look up by UUID or Linear ticket number (e.g. `BUG-42`) |
| `project_summary` | Health snapshot: total active issues, breakdown by status/priority/platform, and activity over the last 7 days |

## OAuth flow

BugSense implements a standard OAuth 2.0 Authorization Code flow with PKCE (S256) for ChatGPT:

1. ChatGPT redirects the user to `GET /api/v1/chatgpt/oauth/authorize` with PKCE parameters.
2. The user signs in with their BugSense email and password on a server-rendered form.
3. On success, the server issues an authorization code and redirects back to ChatGPT.
4. ChatGPT exchanges the code for an access token via `POST /api/v1/chatgpt/oauth/token`.
5. The access token is used in `Authorization: Bearer` headers for MCP requests.

### Scopes

| Scope | Description |
|-------|-------------|
| `projects:read` | List and view projects |
| `issues:read` | List and view issues and comments |
| `reports:read` | View project summaries |

All scopes are granted by default if none are explicitly requested.

### Token lifecycle

- Access tokens are valid for 1 hour (configurable via `CHATGPT_OAUTH_ACCESS_TOKEN_TTL_SECONDS`)
- Authorization codes expire after 10 minutes and are single-use
- Tokens can be revoked server-side

## Well-known endpoints

The backend serves OAuth and MCP discovery metadata:

| Endpoint | Purpose |
|----------|---------|
| `/.well-known/oauth-authorization-server` | OAuth server metadata (endpoints, grant types, PKCE support) |
| `/.well-known/oauth-protected-resource` | MCP resource metadata (resource URL, scopes) |
| `/.well-known/openai-apps-challenge` | OpenAI domain verification |

## Configuration

| Variable | Description |
|----------|-------------|
| `CHATGPT_OAUTH_CLIENT_ID` | Expected OAuth client ID (optional — if set, only this client ID is accepted) |
| `CHATGPT_OAUTH_ACCESS_TOKEN_TTL_SECONDS` | Access token lifetime in seconds (default: `3600`) |
| `OPENAI_DOMAIN_VERIFICATION_TOKEN` | Token for OpenAI domain ownership verification |
